## 2026-05-15 - 新增创建用户账号接口（auth/register）

### 问题现象与根因
- 现象：当前仅有 `auth/login`、`auth/refresh`，缺少最基础的用户账号创建入口，导致无法通过 API 新建普通账号。
- 触发条件：联调或测试需要新建账号时，只能手动改库或依赖初始化种子数据。
- 根因：认证模块初始只覆盖“登录/刷新”路径，未实现注册能力。

### 最终决策与取舍
- 决策：在 `POST /auth/register` 新增最小可用实现，仅接收 `username` 与 `password`，密码做 bcrypt 哈希后入库。
- 取舍 1：当前阶段不接入验证码、短信、邮箱验证等身份校验链路，满足“先可创建账号”的短期目标。
- 取舍 2：保留 DTO 基础校验（`IsString`），不做密码复杂度与用户名格式限制，避免过早引入业务规则。
- 取舍 3：登录时将“无角色账号”的兜底角色从 `admin` 调整为 `user`，避免新账号在未分配角色时获得管理员权限。

### 可复用资产
- 复用资产：`AuthService.register`（账号唯一性检查 + 密码哈希 + ACTIVE 状态创建），可作为后续接入验证码/邀请制的基础流程。
- 测试策略：新增 `test/auth.service.spec.ts` 的注册成功/重名失败用例，作为后续扩展注册逻辑的回归基线。
- 文档资产：新增 Swagger 中文注解与请求/响应 DTO（`RegisterDto`/`RegisterResponseDto`），确保 `/docs` 与 `/docs-json` 可直接联调。

### Skill 与多 Agent 决策记录
- 是否调用 skill：否。
- 不调用原因：任务是小范围单模块改动（`auth` + 单测 + 文档记录），现有项目规范已覆盖，无需额外 skill 发现。
- 是否启用多 agent：否。
- 不启用原因：改动集中且写入边界高度重叠，单 agent 可以更低 token 成本完成并减少冲突风险。

## 2026-05-15 - 语言策略确认（存量 TS 触点迁移到 JS）

### 问题现象与根因
- 现象：需要明确“改到哪个文件，就把哪个文件从 `.ts` 逐步迁移到 `.js`”的执行口径，避免规则理解不一致。
- 触发条件：后续需求持续落在历史 TypeScript 文件时，若没有统一口径，可能出现只改功能不做语言迁移的情况。
- 根因：团队希望推进 JS 化，但需要可落地、低风险的渐进式迁移规则。

### 最终决策与取舍
- 决策：采用“按触点迁移”策略：凡当次任务修改到 `.ts` 文件，优先在当次将该文件迁移为对应 `.js`，并同步完成引用更新与最小回归验证。
- 取舍：不做全量一次性迁移，继续按任务触点逐步替换，以控制改动面和回归成本。
- 例外：若受装饰器元数据、编译链路或类型声明限制暂不可迁移，必须在本文件记录阻塞原因与下一步计划。

### 可复用资产
- 规范资产：`AGENTS.md` 中“语言策略（新增与存量迭代）”作为统一执行标准。
- 执行资产：后续每次迁移都在任务结论里追加“迁移文件、验证范围、遗留阻塞”，形成可追踪迁移清单。

## 2026-05-15 - 注册接口触点迁移（TS -> JS）

### 问题现象与根因
- 现象：`auth/register` 已实现，但文件仍是 TypeScript，不符合“修改当前文件即迁移 JS”的新执行口径。
- 触发条件：本次用户明确要求“先将注册接口改掉”，即先落地一次触点迁移样例。
- 根因：历史代码以 `.ts` 为主，尚处于渐进迁移阶段。

### 最终决策与取舍
- 决策：将注册接口涉及文件迁移到 JS：`auth.controller`、`auth.service`、`dto-login`，并保持接口行为不变。
- 决策：在 `tsconfig.json` 开启 `allowJs`，确保 `tsc` 构建链路识别 `.js` 文件。
- 取舍：本次仅迁移注册接口触点，不扩展到 `auth.module`/`jwt.strategy` 等未改动文件，继续遵循“小步快跑”。

### 可复用资产
- 迁移样例：`src/auth/auth.controller.js`、`src/auth/auth.service.js`、`src/auth/dto-login.js` 可作为后续模块迁移参考模板。
- 验证策略：采用“触点单测 + build”最小闭环（`test/auth.service.spec.ts` + `npm run build`）保证迁移后可运行。

### Skill 与多 Agent 决策记录
- 是否调用 skill：否。
- 不调用原因：任务是局部语言迁移与回归验证，现有项目规范和本地工具已足够。
- 是否启用多 agent：否。
- 不启用原因：迁移文件存在强依赖关系，单 agent 连续修改可减少上下文切换与冲突风险。
