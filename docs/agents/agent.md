## [2026-05-15][规范补全-协作与沉淀]
- 问题现象：项目已有基础协作规范，但缺少“发现问题即沉淀”“官方代码规范统一基线”“skill/多 agent 决策记录”的硬约束。
- 根因分析：规范只定义了接口与工程底线，未覆盖执行过程中的决策日志标准，导致复用信息易丢失。
- 最终方案：在 `AGENTS.md` 增补可执行条款；在本文件建立结构化日志模板，要求每次任务记录问题、决策、skill 和 agent 策略。
- 取舍说明：优先采用轻量模板而非复杂流程系统，降低填写成本，确保能长期执行。
- 公共方法封装：
  - 路径：`src/common/*`（按能力分层，禁止单文件堆叠）。
  - 适用边界：跨模块复用 >= 2 次、与具体业务低耦合的方法。
  - 禁用场景：仅一次性业务逻辑或强依赖单模块上下文的方法。
- 代码规范落地（官方规范）：
  - NestJS：采用官方 module/service/controller/provider 分层与 DI。
  - TypeScript：采用官方类型系统实践，避免 `any`，优先精确类型。
  - Prisma：schema 变更统一走官方 migrate 工作流。
  - DTO 校验：采用 class-validator/class-transformer 官方注解模式。
  - Swagger：采用 @nestjs/swagger 官方注解，保持模型完整可见。
- Skill 决策：
  - 是否调用 `find-skills`：是（后续遇到可复用场景先检索）。
  - 调用的 skill：`find-skills`（策略层面纳入规范）。
  - 调用原因：用于发现可安装或可复用的能力，避免重复造轮子。
  - 不调用原因（如有）：本次为规范落盘任务，未涉及具体外部能力执行。
- 多 Agent 决策：
  - 是否使用多 agent：否。
  - 各 agent 负责范围：N/A。
  - 不使用原因（如单 agent）：本次改动范围小且文件集中，单 agent 更高效，避免额外协作开销。
- 测试与验证：已人工校对规则完整性与可执行性；本次仅文档变更，无运行时逻辑改动。
- 后续可复用结论：后续任务统一按本模板记录，优先沉淀 `docs/skills/*` 与 `src/common/*` 可复用资产。

## [2026-05-15][语言策略变更-JS优先]
- 问题现象：团队希望后续代码尽量使用 JS，不再新增 TS，但仓库当前为 TypeScript 工程。
- 根因分析：若直接要求全量迁移 TS，会引入较高成本与稳定性风险；若无明确边界，执行时易产生歧义。
- 最终方案：规范层面确定“仅新增代码 JS 优先，历史 TS 保持不动”，并建立 TS 例外记录机制。
- 取舍说明：优先保证工程稳定与执行一致性，不做一次性技术栈迁移。
- 公共方法封装：
  - 路径：`src/common/*`。
  - 适用边界：新增 JS 代码中的跨模块公共能力。
  - 禁用场景：仅一次性或强耦合业务逻辑。
- 代码规范落地（官方规范）：
  - NestJS：沿用官方分层与 DI。
  - TypeScript：仅用于存量代码维护，不作为新增默认语言。
  - Prisma：沿用官方 migrate 工作流。
  - DTO 校验：沿用 class-validator/class-transformer 官方模式。
  - Swagger：沿用 @nestjs/swagger 官方注解规范。
  - JavaScript 优先策略：
    - 是否新增 TS 文件：否（本次仅文档变更）。
    - 若新增 TS，业务原因：N/A。
    - 若新增 TS，JS 替代方案不可行原因：N/A。
- Skill 决策：
  - 是否调用 `find-skills`：否。
  - 调用的 skill：N/A。
  - 调用原因：N/A。
  - 不调用原因（如有）：本次为项目内部规范调整，不涉及新增技能能力检索。
- 多 Agent 决策：
  - 是否使用多 agent：否。
  - 各 agent 负责范围：N/A。
  - 不使用原因（如单 agent）：文档改动集中、决策单一，单 agent 效率更高。
- 测试与验证：完成 `AGENTS.md` 与本文件一致性检查，确认无“全量迁移 TS”隐含要求。
- 后续可复用结论：后续任务若新增 TS 文件，必须在记录中写明例外理由与不可替代性。
